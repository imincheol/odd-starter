# 오더 분석: 아키텍처 및 로드맵 관리 체계 정비 (Refine Architecture & Roadmap Management)

## 1. 배경 (Background)
현재 ODD 시스템에서 완료된 태스크가 로드맵(`tasks/roadmap.md`)에 계속 남아있는 문제와, 아카이브(`archive/`) 및 히스토리(`history/`)의 구조적 모호성이 제기되었습니다. 프로젝트가 진행됨에 따라 "완료된 항목"과 "과거의 상태"를 어떻게 관리할지에 대한 명확한 규칙(Spec)이 필요합니다.

## 2. 문제점 (Issues)
1.  **로드맵 비대화**: 완료된 태스크가 `roadmap.md`에 계속 누적되어, 현재 집중해야 할 작업 식별이 어려움.
2.  **아카이브 구조 부재**: `archive` 디렉토리가 단순히 날짜별 플랫 구조로 되어 있어, 무엇이 아카이브 되었는지(태스크인지 로드맵인지) 구분이 모호함.
3.  **명세(Spec) 누락**: 이러한 관리 방침이 `ATLAS.md`나 `books/` 내 문서에 명시되지 않음.
4.  **잔여 파일 문제**: 이전 작업(`05`, `06` 등)의 결과물이 커밋되지 않은 채로 남아있음.

## 3. 목표 (Objectives)
1.  **로드맵 관리 정책 수립**: 로드맵은 "현재와 미래"만 보여주고, 완료된 마일스톤은 아카이브로 이동시키는 정책 확립.
2.  **아카이브 구조 재설계**:
    *   `docs/odd/archive/tasks/{YYYY}/{MM}/{DD}/`: 완료된 태스크 보관.
    *   `docs/odd/archive/roadmaps/{YYYY}/{MM}/{DD}/`: 로드맵 스냅샷 보관.
    *   기존 `history` 디렉토리와의 역할 정립 (시스템 변경 로그는 `history`, 작업물 아카이브는 `archive` 등).
3.  **문서화 (Documentation)**: 변경된 정책을 `ATLAS.md`, `structure.md` 등에 반영.
4.  **정합성 맞추기**: `install.sh` 등 템플릿과 실제 구조의 일치 여부 확인 및 수정.

## 4. 실행 계획 (Action Plan)
1.  **구조 변경 제안 및 확정**: 사용자에게 상기 구조 변경 안을 제시하고 승인 시 수행.
2.  **파일 이동 및 정리**: 현재 `archive` 내 파일들을 새로운 구조(`archive/tasks/...`)로 이동.
3.  **로드맵 정리 (Clean-up)**: `roadmap.md`에서 완료된 항목을 제거하고 최신 스냅샷을 아카이브로 저장.
4.  **문서 업데이트**: `structure.md` 및 `ATLAS.md` 수정.
5.  **커밋**: 방치된 파일들을 포함하여 상태 동기화.
